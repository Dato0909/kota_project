# CLAUDE.md

このファイルはClaudeがこのリポジトリで作業する際のガイドラインです。

---

## 特筆ルール：記事の文体・構成規則

以下のルールは `articles/` 配下の参考資料（官公庁向けコンサルティングレポート）を分析して抽出した、この記事の文体・構成の規則です。記事を作成・編集する際は必ずこのルールに従ってください。

---

### 0. CLAUDE.md の継続的更新ルール

**構成・内容にフィードバックをもらったら、抽象化した学びをこのファイルに反映すること。**

具体的な手順:
1. フィードバックを受けたら、その指摘の背後にある**原則・パターン**を抽象化する
2. 「次も同じ判断ができる」レベルの言葉でルールに落とし込む
3. 既存ルールの修正・追記・新節の追加のいずれかで CLAUDE.md を更新する
4. 変更内容をコミットメッセージに明記して push する

例:「〇〇の節が長すぎる」→「各節は△行以内に収める」というルールを追記する。

---

### 1. ドキュメントの種類と対象読者

- **形式**: 官公庁・企業向けの調査・報告レポート（スライド形式のPDF）
- **対象**: 政策担当者、業界有識者、経営層など専門知識を持つ読者
- **目的**: 調査結果・分析内容・提言を論理的・簡潔に伝える

---

### 2. ページ・スライドの構成

各ページ（スライド）は以下の順序で構成する。

```
[ヘッダー]  章番号・章タイトル / 節番号・節タイトル
            例: 3. ＺＥＢ委員会の準備・運営 / 3.2 調査・分析｜ＺＥＢの普及状況

[サマリー文]  ページの要点を1〜2文で示す（◼ で始める）
              例: ◼ 〜を踏まえ、〜を整理した。

[本文]       箇条書き・表・図のいずれか、または組み合わせ

[フッター]   ページ番号（右下）、著作権表記・会社名（右下）
```

---

### 3. 文体・語調

#### 基本ルール
- **文末**: 「〜した。」「〜である。」「〜のとおりである。」「〜が求められる。」「〜することが重要である。」に統一する
- **語調**: である調（〜である、〜した）を基本とし、格調ある書き言葉を使う
- **一文の長さ**: 60〜80字程度を目安にする。長くなる場合は読点で区切るか、文を分割する
- **主語**: 「本事業では」「本調査において」など、文書・事業を主語にする。「私は」「我々は」は使わない
- **客観性**: 主観的・感情的な表現は一切使わない。事実・データ・論拠に基づいて記述する

#### 多用する表現パターン
| 用途 | 使用する表現 |
|---|---|
| 背景・目的を示す | 〜を踏まえ、〜を背景として、〜に向けて |
| 調査・実施を示す | 〜を実施した、〜を整理した、〜を検討した |
| 付加・補足を示す | 〜するとともに、〜に加え、なお〜 |
| 因果を示す | このため〜、これにより〜、〜ことから |
| 重要度を示す | 特に〜、〜が重要である、〜が求められる |
| 列挙の締め | 〜等、〜など（必ず最後に付ける） |
| 追加事項 | また〜、加えて〜 |

---

### 4. 見出し・ナンバリング

#### 章・節の番号体系
```
1.  大章（章タイトルのみのセクション区切りページを設ける）
1.1 節
1.1.1 項
```

#### 見出しの書き方
- 見出しは**体言止め**にする（例:「調査の背景・目的」「実施概要」「まとめ」）
- 動詞で終わらせない（「〜について説明する」ではなく「〜の概要」）
- 長い見出しには「・」または「〜に向けた〜」の形式を使う

---

### 5. 箇条書きの記号と階層

```
◼  第1階層（大項目、各ページの要点・サマリー文）
•  第2階層（中項目、説明・詳細）
➢  第3階層（小項目、具体例・補足）
⚫  ハイライト・結論のまとめ（囲み枠的な強調、ヒアリング結果のサマリーに多用）
```

- 箇条書き1項目は1〜3行以内に収める
- 体言止め（名詞・動詞の連用形止め）を基本とする
- 各階層をむやみに深くしない（3階層まで）

---

### 6. 表の作成方法

#### 基本構造
- **必ずヘッダー行を設ける**（太字または網掛け）
- セルの内容は簡潔に（箇条書き・体言止め）
- 空欄には「ー」を使用する（N/A・空白・「なし」は使わない）
- 出典がある場合は表の下に「出所）〜より作成」と記載する

#### よく使う表の型
| 型 | 列構成の例 |
|---|---|
| プロセス整理 | 調査ステップ ｜ 目的 ｜ 作業内容 |
| スキルマップ | 項目 ｜ ポイント ｜ 知識 ｜ スキル |
| 比較表 | 手法 ｜ 特徴 ｜ メリット・デメリット |
| 開催実績 | 回数 ｜ 実施日時 ｜ 議事 |
| 調査対象一覧 | No ｜ 企業名 ｜ 業態 ｜ 業種 ｜ 実施日時 |
| 件数集計 | 対象技術・施策名 ｜ 年度別件数 ｜ 累計 |

### 7. 図・ダイアグラムの使い分けと作成方法

#### 7.1 どのビジュアルをいつ使うか

| 伝えたい内容 | 使用するビジュアル |
|---|---|
| 数値の年度別・時系列変化 | 折れ線グラフ・棒グラフ（推移グラフ） |
| 構成比・達成率・普及率 | 円グラフ・積み上げ棒グラフ・割合表示 |
| 複数要素の量を並べて比較 | 棒グラフ（横・縦）・比較表 |
| 手続き・業務・調査のステップ | フロー図（横並び/縦並びボックス＋矢印） |
| 関係者・組織・要素の関係性・役割分担 | スキーム図・関係図 |
| 複数の選択肢・手法・事例の横断比較 | 比較表（行=項目、列=比較軸） |
| 件数・集計データを一覧で整理 | マトリクス表・集計表 |
| 課題・論点・障壁の列挙 | 箇条書き（◼ + •）で十分（図不要） |
| 計画・目標・スケジュールの時系列整理 | ロードマップ（横軸=時間、縦軸=施策） |
| 概念・構造・体系の説明 | 概念図・体系図（ブロック＋矢印） |
| インタビュー・ヒアリング結果の要約 | ⚫ テキストサマリー（カテゴリ見出し＋箇条書き） |

#### 7.2 図を使うか箇条書きで済ませるかの判断基準

- **図を使う**: 関係性・流れ・量の比較・時系列変化など、テキストだけでは構造が伝わりにくい場合
- **箇条書きで済ませる**: 単純な事実の列挙・課題の列挙など、図にしても情報量が増えない場合
- **表を使う**: 複数の属性を複数の対象について整理する場合（行×列の構造がある場合）
- **図と表を両方使う**: 図で全体像を示し、表で詳細を補足する

#### 7.3 図・ダイアグラムの作成ルール

- **サマリー文を必ず前置する**: 図の直前に ◼ で始まるサマリー文を1〜2文置き、図が何を示すかを先に説明する
  - 例: 「◼ 〜について、以下に示す〇〇のステップで実施した。」
  - 例: 「◼ 〜の推移を整理したところ、〜であることが確認された。」
- **出所を必ず記載する**: 図・グラフ・表の下に「出所）〜より、〇〇作成」を付記する
  - 一次データの場合: 「出所）〜データより〇〇作成」
  - 引用の場合: 「出所）〜より抜粋」
  - 複数出所: 「出所）〜より、〜より、〇〇作成」
- **注記を活用する**: データの定義・集計条件・例外事項は「注）」または「※」で図の下に補足する

#### 7.4 グラフ・チャートの記述ルール

- グラフのタイトルは体言止め（例: 「〜の推移」「〜別の割合」「〜件数の比較」）
- 軸・凡例のラベルは簡潔に（単位を括弧で添える: 「（件）」「（%）」「（千万人）」）
- 複数系列がある場合は凡例を必ず付ける
- グラフ内で特筆すべき値には注釈（矢印・吹き出し）を付けることができる

#### 7.5 フロー図・スキーム図の記述ルール

- ボックス（要素）は体言止めで簡潔に記載する
- 矢印の方向で「プロセスの流れ」または「影響・依存関係」を示す
- 関係者を主体にしたスキーム図では、各主体の役割を矢印の横に添える（例: 「支援（施設整備等）」「協力・報告」）
- 複雑になる場合は関係図を分割して複数ページで説明する

---

### 8. 数字・記号の表記ルール

- **年度表記**: 「令和○年度」（和暦）を基本とし、西暦を括弧で添えることもある（例: 令和6年度（2024年度））
- **数字**: 半角数字を基本とする（全角は固有名詞・法令名のみ）
- **パーセント**: 「○○%」（半角・記号）
- **順位・番号**: 「第○回」「①②③」「1. 2. 3.」を使い分ける
- **省略**: 初出は正式名称+略称を示す（例: 「事業承継・引継ぎ支援センター（以下「センター」という。）」）

---

### 9. サマリー文の書き方

各ページ冒頭の1〜2行で「このページで何を伝えるか」を先に示す。以下のパターンを参考にする。

```
◼ 〜について、〜を踏まえ、〜を整理した。
◼ 〜の調査の結果、〜であることが確認された。
◼ 〜に向けて、〜を実施するとともに、〜を行った。
◼ 本事業は、〜を目的として、〜を実施するものである。
◼ 以下には〜の概要を示す。
◼ 〜の推移を整理したところ、〜傾向にあることが確認された。
```

- 結論先行（BLUF: Bottom Line Up Front）で書く
- 詳細は本文（箇条書き・表・図）で補足する

---

### 10. 避けるべき表現

以下の表現は使用しない。

| 避けるべき表現 | 理由 | 代替表現 |
|---|---|---|
| 〜と思います | 主観的・不確か | 〜と考えられる、〜が想定される |
| 〜でしょう、〜かもしれません | 推量・曖昧 | 〜する可能性がある、〜と見込まれる |
| 〜させていただきます（多用） | 過度な謙譲 | 〜する、〜を実施する |
| いい感じ、うまくいく | 曖昧な評価表現 | 適切な、効果的な、〜を達成する |
| ちょっと、少し | 程度が不明確 | 一定程度、軽微な、〇〇%程度 |
| 〜ですが、〜ですので | 話し言葉調 | 〜であるが、〜であるため |
| 〜してください（本文中） | 指示口調 | 〜することが求められる、〜する必要がある |
| 非常に、とても（感嘆的） | 感情的・主観的 | 相当程度、大幅に、顕著に |
| 〜でしょうか？（本文中） | 問いかけ口調 | 記述・断言形式に変換する |

### 11. 全体構成のテンプレート

```
表紙
  └ 発注者名
  └ 事業名（正式名称）
  └ 報告書タイトル
  └ 作成者（会社名・部署名）
  └ 年月

目次
  └ 章・節の一覧とページ番号

本文
  └ 1. 事業の全体像（背景・目的・事業内容）
  └ 2. 〔調査テーマA〕の準備・運営
      └ 2.1 実施概要
      └ 2.2 調査・分析
          └ 2.2.1 〔小テーマ〕
          └ 2.2.2 〔小テーマ〕
          └ 2.2.3 まとめ
      └ 2.3 取りまとめ資料
  └ 3. 〔調査テーマB〕の準備・運営
      └ （同様の構成）

付属資料
```

### 12. まとめ・結論のページ

- 各章末尾に「まとめ」のページを設ける
- 本文で扱った論点を箇条書きで再整理する
- 「今後の課題」「次のステップ」を示す場合は別ブロックで記載する
- 「〜が重要である。」「〜に向けた取組が求められる。」で締めくくる
